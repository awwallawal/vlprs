# Story 15.6: Attention Items & Observation Integration

Status: ready-for-dev

## Story

As a **Super Admin (AG)**,
I want the onboarding pipeline and retirement/deceased verification findings to surface in the existing attention and observation frameworks,
So that the issues come to me on the executive dashboard without needing to navigate to separate pages.

**Origin:** Epic 15, FR33(l) extended, FR87(5) extended. This is the integration layer — wiring Epic 15's findings into the AG's primary view (executive dashboard attention items) and the existing observation engine.

**Dependencies:** Stories 15.3 (monthly onboarding scan — partially implements onboarding lag attention items), 15.4 (onboarding dashboard — drill-down target), 15.5 (retirement verification — provides retirement/deceased action items).

## Acceptance Criteria

1. **Given** approved beneficiaries `NOT_YET_OPERATIONAL` for ≥3 months, **When** the AG views the executive dashboard, **Then** an amber attention item appears: "{count} approved beneficiaries awaiting first deduction across {N} MDAs" with drill-down link to Onboarding Pipeline Dashboard filtered to relevant MDA. *(Note: the detector function `detectOnboardingLag()` is implemented in Story 15.3 Task 3 — this story verifies it surfaces correctly on the dashboard and adds the drill-down URL.)*

2. **Given** approved beneficiaries `NOT_YET_OPERATIONAL` for ≥6 months, **Then** the attention item escalates to gold (review) status with updated wording: "Review recommended — {count} beneficiaries awaiting first deduction for 6+ months."

3. **Given** the onboarding lag attention item, **When** the AG clicks it, **Then** it navigates to `/dashboard/onboarding?status=awaiting&minMonths=3` (Onboarding Pipeline Dashboard from Story 15.4, filtered).

4. **Given** retirees/deceased records in `approved_beneficiaries` with verification status requiring action (NO_RETIREMENT_EVENT, RETIREMENT_GAP, DECEASED_ACTIVE_LOAN from Story 15.5), **When** the AG views the dashboard, **Then** an attention item appears: "{N} retirees/deceased awaiting event recording" with link to the Verification Report.

5. **Given** the FR87(5) "No Approval Match" observation is generated by the observation engine during migration processing, **When** the observation description is constructed, **Then** it includes temporal context from the onboarding pipeline: "Staff {name} in {MDA} has deductions recorded since {first_deduction_month}, but no approval record exists for any batch year. {N} months without committee approval on file."

6. **Given** no approved beneficiaries exist in the system (no committee lists uploaded yet), **When** the dashboard loads, **Then** no onboarding-related attention items appear (graceful empty state — no errors, no "0 beneficiaries" noise).

### "No Approval Match" Observation Enrichment

**Current implementation** (`observationEngine.ts` lines 704-831):
- `detectNoApprovalMatch()` checks migration records against submission data
- Currently generates: "Staff {name} at {MDA} has loan deductions but no matching record in any approved beneficiary list"
- No temporal context (when deductions started, how long without approval)

**Enrichment strategy:**
1. After identifying a no-approval-match staff member, query `migration_records` for their earliest period
2. Check `approved_beneficiaries` for any batch that might contain them (guards against false positives)
3. Add temporal context to description: "since {period}, {N} months"
4. Store in `context` JSONB for downstream consumers

**Guard:** Only enrich if at least one approval batch has been uploaded. Before Epic 15, no approval lists exist — the observation engine shouldn't reference "batch years" when there are none.

### Priority Hierarchy for Epic 15 Attention Items

| Item | Priority | Category | When |
|---|---|---|---|
| Onboarding lag ≥6 months | 10 | review (gold) | Escalated — genuinely late |
| Pending retirement events | 15 | review (amber) | Committee said they're retired, system hasn't recorded it |
| Onboarding lag ≥3 months | 20 | review (amber) | Normal lag threshold |

Lower priority number = higher importance = appears first in the attention items list.
