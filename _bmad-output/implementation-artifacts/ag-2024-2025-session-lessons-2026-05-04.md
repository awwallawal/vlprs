## Lesson 1 — LPC Out is a first-class native column, not just an inferred concept

**Discovered:** 2026-05-04. **Validated:** 2026-05-06 — alias `^(plc|lpc)\s*out$/i` added to `column-map.ts`; native value extracted in build script with cross-MDA-index fallback.

**Discovery:** FINANCE template ships a `PLC OUT` column (typo for "LPC Out" = Last Pay Check Out). PRD has no LPCO concept; FR61 Transfer Out events were the closest analogue. The team had assumed LPC Out would be synthesised from cross-MDA period analysis. Reality: at least one MDA already captures it natively at source.

**Implications for Epic 17:**
- **17.5 person-link-candidates-transfer-handshake-wiring** — engine must read native LPC Out from MDA submissions when present, NOT only infer from cross-MDA timeline. Native value is the source of truth (Agreement 22).
- **17.11 missing-record-detection-mda-prompt** — LPC Out marks the *expected end* of deductions in this MDA. After LPC Out + 2 grace months, missing-record detector should NOT fire.
- **17.22 settlement-pathway-3-event-car-loan-dept-ux** — LPC Out + non-zero outstanding balance = transfer-with-debt = handshake required (not Path 3).
- **17.7 loan-detail-page-unified** — Identity tab should display LPC Out alongside MDA-level history.

**Concrete FR candidate (Addendum 3):**
> **FR-NEW (Addendum 3):** MDA submission templates capture an optional "LPC Out" column. When present, the LPC Out date is authoritative. When absent, system infers from cross-MDA timeline analysis with a confidence flag.

**Existing FRs to amend:** FR61 Transfer Lifecycle — add LPC Out as a filed field. FR101 MDA Beneficiary Ledger — add LPC Out column.

## Lesson 2 — Template defect classes are industrial-scale, not one-off

**Discovered:** 2026-05-04. **Validated:** 2026-05-06 — 25+ alias regexes added to `column-map.ts`; word-fragment join in `header-detect.ts`; defect counts dropped from 24/41 MDAs to ~3 post-fix.

**Discovery:** 24 of 41 MDAs had at least one column-mapper defect. Defect classes were systemic, not random:
- Typos: `OUSTANDING` (missing T), `Comencement` (missing M), `Interst` (missing E), `Monthlly`, `Instalmt`
- Multi-row headers: `INSTALL` (row 2) + `MENT` (row 3) → engine joined as `INSTALL MENT` (with space)
- Abbreviations: `Total Int. paid`, `NO. OF INSTL.PAID`, `OUTSTANDING BAL.`
- Word-order swaps: `BALANCE OUTSTANDING` instead of `OUTSTANDING BALANCE`
- Punctuation variants: `MDA's`, `MDA\`s` (backtick instead of apostrophe)
- Currency suffixes: `Principal (N)`, `Interest (₦)` — block regex match

**Implications for Epic 17:**
- **17.2 port-side-quest-utilities** must absorb every alias from the side-quest. Each defect = a regression test fixture.
- **17.16 idempotency-property-test-framework** — extend with full-corpus invariant: defect counts must not increase across commits.
- **Architectural recommendation:** introduce a **template fingerprint** concept — engine learns each MDA's typical column ordering, validates against drift, flags new variants for review rather than silently dropping.

## Lesson 10 — Fuzzy match has false-positive risk in low-cardinality high-stakes domains

**Discovered:** 2026-05-04. **Validated:** 2026-05-06 — fuzzy-match approach for month-name typos was REPLACED with explicit `MONTH_TYPO_MAP` after `'NEW' → NOV` false positive surfaced in audit. Pattern then extended in Lesson 14 with Yoruba diminutive map (~30 pairs).

**Discovery:** First pass at typo tolerance used Levenshtein-based fuzzy match against month names. Bug: `'NEW'` (in "NEW CAR LOAN TEMPLATE") matched `NOV` with Levenshtein distance 2. Fixed by replacing fuzzy match with explicit typo lookup.

**General principle:** for **low-cardinality** (~12 months, ~63 MDAs) **high-stakes** (drives variance attribution, tier classification) domains, **explicit lookup beats fuzzy matching**. Fuzzy matching is appropriate for **high-cardinality** (thousands of names) **lower-stakes** (one of many signals) domains.

**Implications for Epic 17:**
- **17.4 person-identity-service** — uses Jaccard + Levenshtein for name matching. Fine for high-cardinality, but should ALSO carry an explicit known-variants table maintained alongside the algorithm. See Lesson 14 for the concrete Yoruba diminutive map.
- **17.2 port-side-quest-utilities** — typo maps should be editable JSON (not hardcoded TS) so AG/Dept Admin can add typos discovered in the field without a deploy.

## Lesson 11 — Mid-sheet period split is industrial-scale (NEW 2026-05-06)

**Discovered:** 2026-05-06. **Validated:** 2026-05-06 — `findPeriodMarkers` + `getPeriodForRow` shipped; full-corpus diagnostic identified **656 sheets** across the corpus with multi-period stacking.

**Discovery:** Many MDA submissions stack multiple months in a single sheet, separated by "CAR LOAN FOR THE MONTH OF X" sub-headers at distinct row positions. The original parser assumed `1 sheet → 1 period`. Examples:
- `EDUCATION JAN_25_DEC_25_*.xlsx` — 12 months stacked in each sheet (582 markers detected)
- `ENERGY Sheet2 / Sheet3` — full year per sheet
- `LANDS car returns 2024.xlsx` — 528 markers across 12 mid-split sheets

**Locked-phrase regex avoids false positives** — the unambiguous phrase `(?:CAR|VEHICLE|MOTOR\s*VEHICLE|VEHINCLE)\s*LOAN\s*(?:RETURNS?|PAYMENT|...)?\s*FOR\s*(?:THE\s*)?MONTH\s*OF\s*<MONTH>,\s*<YEAR>` matches only legitimate section headers, not data-row date columns.

**Implications for Epic 17:**
- **17.13 upload-pipeline-integration-content-validation** — must detect mid-sheet splits at ingest time, NOT silently process them. Officer should confirm intent.
- **17.2 port-side-quest-utilities** — port `findPeriodMarkers` + period-block-splitter as first-class capability of the main app's parser.
- **17.16 idempotency-property-test-framework** — fixtures must include known-mid-sheet-split files (EDUCATION, LANDS, HOUSING, ENERGY, AGRIC).

## Lesson 12 — Stale copy-paste period markers (template artifact)

**Discovered:** 2026-05-06. **Validated:** 2026-05-07 — two independent fixes shipped: (a) `MAX_PERIOD_MARKER_COL = 30` filter in `findPeriodMarkers`; (b) `manifestSheet.period.source === 'sheet'` authoritative rule in `car-loan-parse.ts`.

**Discovery:** MDA officers commonly create templates by copying a previous month's file and editing the title. **They forget to update copy-paste artifacts in non-obvious cells:**
- LGA `auditor_general_LG JAN-MAY 2025` had `JANUARY 2024` markers at columns OM4 / OK3 / OP4 (col 400+!) — far-right cells the officer never sees
- AGRIC 2024 OCTOBER sheet had a leftover `AUGUST 2024` marker at row 165 col 1 — body-cell text someone forgot to update

**Without the fixes** these stale markers triggered mid-sheet-split logic and silently misattributed entire CDU sub-sections to wrong months. **The naive fix (just trust body markers) caused regressions of −417 records on LGA and −38 records on CDU.**

**Two-layer fix:**
1. **Spatial filter** — `MAX_PERIOD_MARKER_COL = 30`. Real period section-headers always live in cols A-AA; col 400+ is template artifact.
2. **Authority rule** — when `manifestSheet.period.source === 'sheet'` (sheet name unambiguously names the period like "OCTOBER 2024"), the manifest period is authoritative. Body markers cannot override. Mid-sheet split logic only fires when sheet-name is generic (`Sheet1`/`Sheet2`/etc.).

**Implications for Epic 17:**
- **17.13 upload-pipeline-integration** — at ingest, detect stale markers and flag the file: "officer, your sheet header at row N still says X but the sheet name says Y; please confirm."
- **MDA officer dashboard (15.0e)** — add a "template freshness" check at submit time.
- **17.10 most-likely-explanation-suggestion-engine** — STALE_TEMPLATE_MARKER as a new variance class.

## Lesson 13 — Yoruba name-order swap (NEW 2026-05-07)

**Discovered:** 2026-05-07 (canary: AWODELE EUNICE ADEOLA — Awwal's BIR question). **Validated:** 2026-05-07 — token-sort canonicalization in `staffKeyOf` collapses both spellings.

**Discovery:** Same person submitted under different name orders by the same MDA across different months:
- `AWODELE ADEOLA EUNICE` (Yoruba style: surname/middle/first)
- `AWODELE EUNICE ADEOLA` (English style: surname/first/middle)

**Canary:** `AWODELE ADEOLA EUNICE ↔ AWODELE EUNICE ADEOLA` and `FADIPE COMFORT FOLASADE ↔ FADIPE FOLASADE COMFORT` (Accountant General).

**Implications for Epic 17:**
- **17.4 person-identity-service** — token-sort is mandatory normalization step. Order-independent matching is the baseline.
- **17.4 PIS extension** — when token-sort merges 2+ records, validate that financials agree (else namesake split — see Lesson 17 / namesake guard).

## Lesson 15 — Ghost rows from secondary tables (NEW 2026-05-07)

**Discovered:** 2026-05-07 (canary: AWODELE / AWOFOLAJU sparse rows). **Validated:** 2026-05-07 — `pickLatestRecord` tie-break by `completenessScore` ensures sparse "ghost" records yield to canonical full records.

**Discovery:** Some MDA templates have **secondary tables** within the same sheet — typically a summary list at the bottom or a comparison table. These tables have only a subset of columns populated (e.g., BIR Sheet1/Sheet2/Sheet3 ghost rows have only `Monthly Deduction` populated; all other financial columns are null).

The parser correctly extracts these as records (they have a name, they have a deduction value), but they're sparse — they pollute aggregation if treated as full records.

**Fix:** when grouping records under a staffKey, the most-complete record wins (not just the latest by period). This silently absorbs ghost rows into the canonical full record.

**Long-term fix (Epic 17):** detect "secondary table" pattern at parse time — a row with only 1 non-null financial column where the rest of the sheet's rows have many — and skip or tag as `PARSER_NOISE`.

**Implications for Epic 17:**
- **17.13 upload-pipeline-integration** — detect-and-skip secondary tables at ingest. Or warn officer that the table is unexpected.
- **17.2 port-side-quest-utilities** — `pickLatestRecord` with completeness tie-break is the canonical pattern for picking representative records from a person-period bucket.

## Lesson 17 — Tie-break by completeness in dedup/aggregation (NEW 2026-05-07)

**Discovered:** 2026-05-07. **Validated:** 2026-05-07 — `pickLatestRecord` updated; `dedupMap` resolution preserves more-complete record on conflict.

**Discovery:** When two records share the same dedup key (or staffKey), the LATEST one shouldn't always win. A sparse record from a later period can override a complete record from an earlier period. The right rule is: **pick the latest record AT each completeness tier; a record with full data outranks a record from a later period with sparse data**.

**Fix:** `completenessScore(r) = count of non-null financial fields`. Use as secondary sort after period.

**Implications for Epic 17:**
- **17.12 person-reconciliation-pass-prp** — PRP must use completeness tie-break. When multiple records exist for the same (person, period), the more-complete one is canonical for snapshot purposes.
- **17.15 monthly-dashboard-snapshots** — snapshot row picker uses same rule.

## Lesson 18 — Namesake guard via principal agreement (NEW 2026-05-07)

**Discovered:** 2026-05-07 (canary: AMOSUN B.K vs AMOSUN K.B in ACCOS — different principals, different people). **Validated:** 2026-05-07 — namesake guard in `build-ag-2024-2025-report.ts` splits records by principal-bucket when staffKey collapses across distinct loan amounts.

**Discovery:** Token-sort + diminutive expansion can over-merge — different people with similar tokens collapse to the same staffKey. Examples:
- `AMOSUN B.K` (₦750K loan) ↔ `AMOSUN K.B` (₦450K loan) — different people, different principals
- Same name but with different sequential loans: `AWOFOLAJU SAMUEL TEMITOPE` had Loan 1 (₦600K, 50mo, Jun 2022→mid-2024) AND Loan 2 (₦750K, 40mo, Oct 2024→ongoing)

**Fix — three-rule namesake guard:**
1. If only 1 distinct non-null principal → keep merged
2. If multiple principals within 10% → keep merged (MDA-computed-variance class)
3. Otherwise → split into per-principal-bucket sub-keys; null-principal records attach to the largest bucket

For AG report purposes: **same staff with sequential loans correctly produces 2 rows (one per loan instance)** — the AG schema is per-loan, not per-staff.

**Implications for Epic 17:**
- **17.4 person-identity-service** — namesake guard is the reverse of the canonicalizer/diminutive expansion. Both must coexist:
  - Canonicalizer (Yoruba 9-rule) + diminutive map → COLLAPSE same person
  - Namesake guard (principal agreement, employee-no agreement, period-overlap) → SEPARATE different people
- **PIS algorithm:** explicit policy needed for "same name, different loans" — this is NOT a namesake (same person, sequential loans). Use principal + period + tenure to disambiguate.
- **17.10 most-likely-explanation-suggestion-engine:** include "sequential loans for same staff" as a narrative.

## Lesson 19 — Year-aggregate submissions are a valid pattern (NEW 2026-05-07)

**Discovered:** 2026-05-07 (canary: FINANCE — Awwal's flag that FINANCE submitted full 2024+2025 year-end snapshots, but engine showed Tier B with only 2024-12 + 2025-12 + a 2024-00 ghost bucket). **Validated:** 2026-05-07 — `YEAR_AGGREGATE_PATTERN` in `period-extract.ts` recognizes "YYYY Staff Car Loan", "YYYY Public Office Holder Car Loan" and similar phrasing, attributing as YYYY-12 (year-end snapshot).

**Discovery:** Some MDAs submit a single sheet per year as a year-end aggregate, NOT a monthly submission. FINANCE's `Finance_Car loan Return 2024 2025.xlsx` has 4 sheets:
- `2024 Staff Car Loan` — 66 rows = 78 staff full year-end state
- `2025 Staff Car Loan` — 62 rows
- `2025 Public Office Holder Car L` — 27 rows (note Excel sheet-name truncation at 31 chars cuts "Loan" to "L")

**Critically:** these sheets carry the AG schema columns natively (Commencement Date, Completion Date, PLC Out, No of Installment, Principal, Total Interest, Total Loan, etc.). FINANCE submitted EXACTLY what the AG asked for — better than monthly format for AG year-end purposes.

The engine's monthly-submission assumption mis-classified this as Tier B (1 month coverage) when it should be conceptually Tier A-equivalent ("full year via single year-end snapshot").

**Implications for Epic 17:**

- **17.17 dual-truth-dashboard-rendering** — tier classification needs a "year-aggregate" tier indicator distinct from monthly tiers (or rename existing tiers to be format-aware).
- **AG report cover letter:** explicitly note when an MDA submitted year-aggregate vs monthly. The "1 month coverage" framing is misleading for FINANCE-class submissions.

**Discovered:** 2026-05-07 (Awwal's challenge: "I thought you ingested data from C:\Users\DELL\Desktop\qa before you began your work"). **Validated:** 2026-05-07 — content-hash audit of qa_qa shows all 437 files have content-identical twins in drop zone (under different names from earlier triage); WAKEUP corrected to remove "retired" misframing.

**Investigation revealed:**
- 437 files in qa_qa
- All 437 files have content-identical twins in `docs/Car_Loan/` (verified by SHA-256 hash)
- The earlier Phase 0–3 triage (2026-04-17) had copied them all, often with `qaqa_*` prefix transformations
- WAKEUP's "retired" label was technically correct (no NEW files belong there) but ambiguously phrased — read as "ignored/uningested" rather than "fully-ingested-already-don't-add-more"

**Lesson:** "retired" status of any source archive must be backed by **content-hash audit at the time of marking**, not just folder-level decisions. The audit must be re-runnable as a CI gate — if a new file appears in a "retired" folder OR if a drop-zone file's content hash doesn't match its claimed retired-archive twin, surface as anomaly.

**Tool produced this session:** `scripts/legacy-report/import-qa-archive.ts` — content-hash dedup + folder-path MDA inference + `qaqa_<mda>_` filename prefix. Ready to run if future archive imports needed.

**Implications for Epic 17:**

- **17.16 idempotency-property-test-framework** — extend invariants to include "every file in retired-archive matches a drop-zone twin by content-hash."
- **WAKEUP-as-living-doc principle:** any "retired" / "do not use" marker on a folder needs an annotated reason + verification command. Tom's-eye check: a future agent reading WAKEUP cold should know HOW to verify the claim, not just trust it.
