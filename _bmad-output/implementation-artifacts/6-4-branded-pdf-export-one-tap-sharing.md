# Story 6.4: Branded PDF Export & One-Tap Sharing

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As any **admin user**,
I want to export any report as a branded PDF with the Oyo State Government crest and share it with one action,
so that reports are suitable for official circulation to the Commissioner, Governor, or auditors.

## Acceptance Criteria

### AC1: PDF Export (FR53)

**Given** any generated report (Executive Summary, MDA Compliance, Variance, Loan Snapshot, Weekly AG)
**When** the user clicks "Export as PDF"
**Then** the server generates a branded PDF via `@react-pdf/renderer` containing:

- Oyo State Government crest
- Official formatting with VLPRS branding
- Generation date
- VLPRS reference number
- The complete report content

**And** the PDF downloads to the user's device

### AC2: One-Tap PDF Sharing (FR54)

**Given** a generated report on the reports page
**When** a Super Admin or Department Admin clicks the "Share" action
**Then** they can:

- Download the PDF (same as AC1)
- Trigger an email send with the PDF attached — one-tap from report view

### AC3: PDF Output Quality

**Given** the PDF output
**When** viewed on any device or printed
**Then** the formatting is consistent, professional, and suitable for Commissioner/Governor circulation

## Dev Notes

### Architecture: Server-Side PDF Generation

All PDF generation happens on the server via `@react-pdf/renderer` v4.3.2. The client requests a PDF endpoint, the server generates the buffer, and returns it as a download. No client-side PDF libraries in the bundle.

```
Client clicks "Download PDF"
  → GET /api/reports/{type}/pdf?{params}
  → Server: generate report data (reuse existing report services)
  → Server: lazy-import PDF generator
  → Server: renderToBuffer(<ReportPdf data={data} meta={meta} />)
  → Server: res.set({ Content-Type, Content-Disposition, Content-Length })
  → Server: res.send(pdfBuffer)
  → Client: blob → createObjectURL → download link → click → cleanup
```

### PDF Route Pattern — Follow traceReportRoutes.ts

```typescript
// Pattern from traceReportRoutes.ts — replicate for each report type
router.get('/reports/executive-summary/pdf', ...adminAuth, auditLog, async (req, res) => {
  // 1. Validate query params (same as JSON endpoint)
  const params = executiveSummaryQuerySchema.parse(req.query);

  // 2. Generate report data (reuse existing service)
  const data = await generateExecutiveSummaryReport(req.mdaScope, params);

  // 3. Lazy-import PDF generator
  const { generateExecutiveSummaryPdf } = await import('../services/executiveSummaryPdf.js');

  // 4. Build metadata
  const meta: PdfReportMeta = {
    referenceNumber: generateReferenceNumber(),
    generatedAt: new Date().toISOString(),
    generatedBy: req.user.fullName,
    reportTitle: 'Executive Summary Report',
    reportSubtitle: params.periodYear ? `Period: ${params.periodYear}-${params.periodMonth}` : 'Current Period',
  };

  // 5. Generate PDF buffer
  const pdfBuffer = await generateExecutiveSummaryPdf(data, meta);

  // 6. Send response
  const dateStr = new Date().toISOString().slice(0, 10);
  res.set({
    'Content-Type': 'application/pdf',
    'Content-Disposition': `attachment; filename="vlprs-executive-summary-${dateStr}.pdf"`,
    'Content-Length': String(pdfBuffer.length),
  });
  res.send(pdfBuffer);
});
```

### Email Sharing Pattern — Follow email.ts

Extend `apps/server/src/lib/email.ts` with a new function. Follow the established pattern:

1. Check `isEmailConfigured()` — if no RESEND_API_KEY, log to console (dev fallback)
2. Lazy-init Resend client via `getResendClient()`
3. Fire-and-forget with error logging (don't crash on email failure)
4. Attachment via Resend API:

```typescript
await resend.emails.send({
  from: env.EMAIL_FROM,
  to: recipientEmail,
  subject: `VLPRS Report: ${reportTitle}`,
  html: reportShareEmailHtml({ reportTitle, coverMessage }),
  attachments: [{
    filename: pdfFilename,
    content: pdfBuffer,  // Resend accepts Buffer directly
  }],
});
```

### Frontend Download Pattern — Follow useDownloadTracePdf

The established pattern in `apps/client/src/hooks/useTraceReport.ts` uses `useMutation` for PDF downloads:

1. Read auth token at call time via `useAuthStore.getState()` (not render time — avoids stale token)
2. `fetch()` with Authorization header
3. `res.blob()` → `URL.createObjectURL()` → create `<a>` element → click → cleanup

**Create a generic `useDownloadReportPdf(reportType, params)` hook** that follows this exact pattern but accepts any report type and query params.

### Share Report Flow

```
User clicks "Share" button on any report
  → ShareReportDialog opens
  → User enters recipient email + optional cover message
  → User clicks "Send"
  → POST /api/reports/share { reportType, recipientEmail, coverMessage, reportParams }

  → Server: send email with PDF attachment (via Resend)
  → Server: return { success: true }

```

- **2026-03-29 — Production Deploy Fix:** Server crashed on startup with `ENOENT: no such file or directory, open '/app/assets/oyo-crest.png'`
  - **Root cause:** `reportPdfComponents.tsx` reads the Oyo State crest PNG via `fs.readFileSync` at module load time. tsup bundles JS but does not copy binary assets. The Dockerfile production stage only copied `dist/` and `drizzle/`, missing `src/assets/` where the PNG lives. The bundled chunk's `__dirname` resolved to `/app/dist/`, so `../assets/oyo-crest.png` pointed to `/app/assets/` which didn't exist in the container.
  - **Fix:** Added `COPY --from=build /app/apps/server/src/assets ./assets` to `Dockerfile.server` production stage.
  - **Retro note:** Any future static assets (images, templates, fonts) added to the server need a matching Dockerfile COPY line. The dev agent should flag non-JS assets in story File Lists as requiring Docker awareness. tsup will never bundle them.
